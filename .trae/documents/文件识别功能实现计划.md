# 文件识别功能实现计划

## 一、功能需求
1. 将"记一笔"功能中的"文件上传"改为"文件识别"
2. 实现文件识别功能，能够从各类文件（笔录、证据、协议等）中提取关键信息
3. 设计API接口，预留未来集成第三方服务的能力
4. 预留API key配置，方便后续添加服务

## 二、技术方案

### 1. 前端修改
- 修改CaseDetail.tsx中的按钮文本和提示信息
- 调整文件上传逻辑，添加文件识别状态显示
- 优化用户体验，显示识别进度和结果

### 2. 后端扩展
- 扩展Evidence模型，添加识别结果字段
- 实现文件识别API接口
- 设计文件识别服务，支持多种文件类型
- 预留第三方服务集成接口

### 3. 数据模型扩展
- 在Evidence模型中添加：
  - `recognizedContent`: 识别后的文本内容
  - `recognizedKeyInfo`: 提取的关键信息
  - `recognitionStatus`: 识别状态（pending, processing, completed, failed）
  - `recognitionTime`: 识别时间

### 4. API设计
- **POST /api/evidence/recognize**: 文件识别接口
- **GET /api/evidence/:id/recognition**: 获取识别结果
- **PUT /api/evidence/:id/recognition**: 重新识别文件

### 5. 第三方服务集成
- 支持的服务：百度OCR、阿里云OCR、腾讯云OCR等
- 配置管理：在环境变量中添加API key
- 可切换的识别引擎

## 三、实现步骤

### 1. 前端实现
- 修改CaseDetail.tsx中的按钮文本为"文件识别"
- 更新提示信息，明确文件识别功能
- 添加识别进度显示和结果预览
- 优化文件上传和识别的用户体验

### 2. 后端实现
- 扩展Evidence模型，添加识别相关字段
- 实现文件识别API接口
- 设计文件识别服务，支持本地处理和第三方服务调用
- 配置环境变量，预留API key设置

### 3. 集成测试
- 测试文件识别功能的完整流程
- 验证不同文件类型的识别效果
- 测试API接口的稳定性
- 验证配置管理的灵活性

## 四、技术选型
- **前端**: React + Ant Design
- **后端**: Node.js + Express + TypeScript
- **文件处理**: multer (文件上传), sharp (图像处理)
- **OCR服务**: 预留接口，支持百度、阿里、腾讯等第三方服务
- **配置管理**: dotenv

## 五、预期效果
- 用户可以上传文件并自动识别其中的关键信息
- 识别结果自动填入"记一笔"的文本框
- 系统支持多种文件类型的识别
- 预留第三方服务集成接口，方便后续扩展
- 提供API key配置，便于服务激活和切换
